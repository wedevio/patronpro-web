# Google Meet creation through Google Calendar API

Project: PatronPro onboarding automation

Date: 2026-06-12
Artifact status: current implementation research note

## Decision

Use Google Calendar API Meet creation only as a future organizer-owned adapter. Keep GHL as appointment source of truth and keep the universal ICS/calendar-link generator as the default client invite path.

This bead produced a dry-run-only Python skeleton. It does not call Google, does not create events, does not send attendee updates, does not touch GHL, and does not persist secrets.

## Required Insert Shape

Endpoint:

```text
POST https://www.googleapis.com/calendar/v3/calendars/{calendarId}/events
```

Required query posture:

```json
{
  "conferenceDataVersion": 1,
  "sendUpdates": "none"
}
```

Important behavior:

- `conferenceDataVersion` must be exactly `1` for conference creation support.
- Google Meet creation is requested with `conferenceData.createRequest`.
- Use `conferenceData.createRequest.conferenceSolutionKey.type = "hangoutsMeet"`.
- `conferenceData.createRequest.requestId` belongs to the conference creation request. Reusing it against the same organizer/calendar can cause the repeated conference creation request to be ignored.
- `events.insert` often returns Meet data synchronously, but implementation should verify `conferenceData.createRequest.status.statusCode` and `conferenceData.entryPoints` instead of assuming success.
- `sendUpdates=none` is not a safety boundary; it can still create a live calendar event. The safety boundary in this bead is no network execution.

## OAuth And Credentials

PoC path:

1. Create/select Google Cloud project.
2. Enable Google Calendar API.
3. Configure OAuth consent/branding for internal test use.
4. Create Desktop app OAuth client for local testing.
5. Store `credentials.json` and `token.json` outside the repo.
6. Use write-capable Calendar scopes for creation testing, not the readonly scope from the quickstart.
7. Delete any old `token.json` when scopes change.

Future server path:

- Use Web application OAuth client and encrypted per-user token storage.
- Treat Calendar write scopes as potentially requiring Google's external app verification before customer-facing rollout.
- Do not assume service accounts can write arbitrary calendars; service accounts need an approved Workspace domain-wide delegation model.

## Dry-Run Skeleton

Script:

```text
dev/agents/artifacts/script/onboarding-automation/google_meet_calendar_poc.py
```

Validator:

```text
dev/agents/artifacts/script/onboarding-automation/validate_google_meet_calendar_poc.py
```

The script emits redacted JSON for the future `events.insert` request. It validates:

- Offset-aware start/end datetimes.
- IANA timezone.
- End after start.
- `conferenceDataVersion == 1`.
- `sendUpdates` values and explicit approval for non-`none`.
- Dry-run request ID is not a literal placeholder.
- Live mode refuses unsafe execution flags before any Google import.
- Attendee email redacts to `***@***` by default.
- PatronPro/GHL correlation fields live under `extendedProperties.private`.

## Live Mode Boundary

Live mode remains out of scope. A future task must not execute Google calls unless all gates are present:

- `--execute`
- `--confirm-live`
- `--credentials-path` outside repo
- `--token-path` outside repo
- `PATRONPRO_GOOGLE_MEET_LIVE_TOKEN` set locally
- `--allow-send-updates` when `sendUpdates` is `all` or `externalOnly`

The current script still exits non-zero after these gates because provider execution is intentionally not implemented in this bead.

## RLM/Commit Safety

Store only redacted payloads. Never commit or store:

- Google `credentials.json`
- Google `token.json`
- OAuth refresh/access tokens
- Google response headers
- cookies/localStorage/session headers
- full attendee emails from real clients

Rollback for this bead should be a normal git revert of the checkpoint commit.
